### MUI란

MUI(Material UI)는 구글의 Material Design을 기반으로 만든 React 컴포넌트 라이브러리다. 수백 개의 UI 컴포넌트가 디자인 토큰과 연결되어 있고, 피그마 커뮤니티에서 무료로 사용할 수 있는 디자인 파일도 제공한다. 실무에서 가장 많이 쓰이는 오픈소스 디자인 시스템 중 하나다.

[Material UI for Figma](https://www.figma.com/community/file/912837788133317724) 파일을 처음 열었을 때 압도되었다. 컴포넌트 페이지, 토큰 페이지, 스타일 가이드 페이지가 따로 있고, 버튼 하나에도 contained, outlined, text variant가 있고, 거기에 색상 3종, 크기 3종, 상태 2종이 조합된다. 경우의 수만 따지면 수십 가지다.

### Alert 컴포넌트 분석

Alert는 사용자에게 중요한 정보를 전달하는 컴포넌트다. MUI의 Alert는 네 가지 severity(심각도) Variant를 가진다.

각 severity마다 배경색, 테두리색, 아이콘, 텍스트 색이 모두 토큰으로 연결되어 있다. error → warning → info → success 순서로 심각도가 낮아진다.

Alert 컴포넌트를 클론하면서 가장 먼저 한 것은 Variant를 error / warning / info / success로 나누는 것이었다. 색을 먼저 정하고 싶었는데, MUI 원본을 보니 색이 아니라 severity(심각도)가 먼저였다. 그 severity에 따라 배경색, 테두리색, 아이콘, 텍스트 색이 모두 토큰으로 연결되어 있었다.

"빨간색 알럿"이 아니라 "error 상태의 알럿"이라는 식으로 생각하는 것이 디자인 시스템의 언어였다.

Button은 contained, outlined, text 세 가지 variant에 primary, secondary, error 색상이 조합되고, 거기에 small, medium, large 크기와 enabled, disabled 상태까지 붙는다. 이 모든 것을 Variant Property로 정의해 두면 인스턴스 하나에서 드롭다운으로 전환할 수 있다.

직접 만들면서 왜 컴포넌트를 처음에 잘 설계해야 하는지 다시 느꼈다. 나중에 "outlined 버튼에 hover 상태 추가해 주세요"라는 요청이 오면, 컴포넌트가 잘 설계되어 있으면 한 번만 수정하면 된다. 그렇지 않으면 화면 전체를 뒤져야 한다.

### Checkbox 컴포넌트 분석

Checkbox는 checked / unchecked / indeterminate 세 가지 상태와 disabled 상태를 갖는다.

<svg width="100%" viewBox="0 0 640 100" xmlns="http://www.w3.org/2000/svg">
  <!-- checked -->
  <rect x="20" y="30" width="20" height="20" rx="3" fill="#1976D2"/>
  <text x="26" y="45" font-size="12" font-weight="600" fill="#fff">✓</text>
  <text x="48" y="44" font-size="11" fill="#2C2C2A">Checked</text>

  <!-- unchecked -->
  <rect x="160" y="30" width="20" height="20" rx="3" fill="none" stroke="#888780" stroke-width="1.5"/>
  <text x="188" y="44" font-size="11" fill="#2C2C2A">Unchecked</text>

  <!-- indeterminate -->
  <rect x="320" y="30" width="20" height="20" rx="3" fill="#1976D2"/>
  <line x1="325" y1="40" x2="335" y2="40" stroke="#fff" stroke-width="2.5"/>
  <text x="348" y="44" font-size="11" fill="#2C2C2A">Indeterminate</text>

  <!-- disabled -->
  <rect x="480" y="30" width="20" height="20" rx="3" fill="#D3D1C7"/>
  <text x="508" y="44" font-size="11" fill="#888780">Disabled</text>

  <text x="320" y="85" text-anchor="middle" font-size="10" fill="#888780">Required* 속성으로 필수 입력 표시 가능 · Label 속성으로 텍스트 직접 편집 가능</text>
</svg>

Checkbox에서 처음 알게 된 상태가 있었다. Indeterminate — 체크박스가 부분적으로 선택된 상태다. 예를 들어 "전체 선택" 체크박스에서 일부만 선택되어 있을 때 표시된다. 완전히 체크된 것도 아니고 완전히 해제된 것도 아닌 중간 상태.

이 상태를 모르면 설계에서 빠뜨리게 된다. MUI를 뜯어보면서 내가 몰랐던 UI 상태들이 꽤 있다는 걸 알게 되었다.

---

MUI를 그냥 가져다 쓰는 것과 직접 뜯어서 만들어 보는 건 완전히 다른 경험이었다. 토큰이 어떻게 연결되어 있는지, 각 컴포넌트에 어떤 Property가 있는지, 왜 이런 구조로 설계했는지가 손으로 만들어 보면서 비로소 이해되었다. 좋은 디자인 시스템을 모방하는 것이 가장 빠른 공부 방법이라는 걸 이번 실습에서 확실히 느꼈다.

> [!tip]
> MUI 같은 오픈소스 디자인 시스템을 뜯어보는 게 아토믹 디자인을 이해하는 가장 빠른 방법이다. 누군가 이미 수천 개의 상태를 설계해 두었으니까.
